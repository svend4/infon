## A. Научная база: алгоритмы конвертации изображений в текст

Терминальный видеокодек TVCP стоит на плечах десятилетий исследований в области text-based image rendering. Понимание этих алгоритмов критически важно для оптимизации кодека .babe.

### A.1. Базовый алгоритм (Luminance Mapping)

1. Изображение переводится в **grayscale** — каждый пиксель получает значение яркости от 0 (чёрный) до 255 (белый)
2. Изображение разбивается на **блоки** (8×16 пикселей — примерный размер одного символа в моноширинном шрифте)
3. Для каждого блока вычисляется **средняя яркость**
4. Яркость **маппится** на символ из заранее отсортированной строки — **ramp** (шкалы плотности)

### A.2. Шкалы плотности символов (Character Density Ramps)

Ключевой элемент. Символы сортируются по тому, сколько пикселей они «закрашивают» при рендере.

**Короткая (10 символов):**
```
@%#*+=-:. 
```

**Стандартная (70 символов, Paul Bourke):**
```
$@B%8&WM#*oahkbdpqwmZO0QLCJUYXzcvunxrjft/\|()1{}[]?-_+~<>i!lI;:,"^`'. 
```

**Формула точного измерения плотности:**
```
density(char) = count(black_pixels) / total_pixels_in_glyph
```

Можно отрендерить каждый символ в bitmap и автоматически посчитать процент закрашенных пикселей — это даёт объективную шкалу, не зависящую от шрифта.

### A.3. Продвинутые алгоритмы

**Structure-based matching (сопоставление по структуре):**

Вместо средней яркости анализируется *форма* блока — направление линий, краёв. Символ `/` подбирается для диагональных краёв, `|` для вертикальных, `_` для горизонтальных. Даёт гораздо более чёткие контуры.

> **Релевантность для TVCP:** Может использоваться как дополнительный режим «edge-aware encoding» для сцен с высокой контрастностью (документы, чертежи на экране).

**Edge detection (обнаружение краёв):**

Фильтры Собеля или Кэнни для нахождения контуров, затем контуры рисуются символами `/`, `\`, `|`, `_`, `-`.

**Собель-операторы:**
```
Gx = [[-1, 0, +1],    Gy = [[-1, -2, -1],
      [-2, 0, +2],          [ 0,  0,  0],
      [-1, 0, +1]]          [+1, +2, +1]]

gradient_magnitude = sqrt(Gx² + Gy²)
gradient_direction = atan2(Gy, Gx)
```

> **Релевантность для TVCP:** Может служить основой для режима «screen sharing» — при показе экрана/документов edge-based рендеринг лучше передаёт текст и UI-элементы, чем luminance mapping.

**Dithering (диффузия ошибки, Floyd-Steinberg):**

Если блок чуть темнее, чем выбранный символ, «ошибка» передаётся соседним блокам. Это даёт плавные градиенты.

```
Матрица распределения ошибки Floyd-Steinberg:
         * 7/16
  3/16 5/16 1/16
```

> **Релевантность для TVCP:** Полезно для сцен с плавными градиентами (лица, освещение). Может быть опцией кодека: `--dither=floyd-steinberg`.

**Метод SSIM (Structural Similarity Index):**

Сравнение не просто яркости, а структурного сходства между блоком изображения и рендером каждого символа. Даёт наилучшее качество, но самый CPU-затратный.

```
SSIM(x, y) = (2μₓμᵧ + c₁)(2σₓᵧ + c₂) / ((μₓ² + μᵧ² + c₁)(σₓ² + σᵧ² + c₂))
```

> **Релевантность для TVCP:** Может использоваться при офлайн-рендеринге (запись звонка → высококачественное ASCII-видео для шеринга).

## B. Полная карта Unicode Block Elements

### B.1. Элементы, используемые в TVCP

```
█ ▓ ▒ ░       — блоки с убывающей плотностью (shading)
▀ ▄ ▌ ▐       — половинные блоки (верх, низ, лево, право)
▖ ▗ ▘ ▝       — четвертные блоки (отдельные квадранты)
▛ ▜ ▙ ▟       — три четверти
┌ ┐ └ ┘ ─ │   — линии для рамок
╔ ╗ ╚ ╝ ═ ║   — двойные линии
╭ ╮ ╰ ╯       — скруглённые углы
```

### B.2. Разрешение по типу рендеринга

| Метод | Субпикселей / символ | Эффективное разрешение (80×30 терминал) |
|---|---|---|
| Символы ramp (@ # + .) | 1 | 80×30 = 2,400 |
| Half-blocks (▀▄) | 2 (верх + низ) | 80×60 = 4,800 |
| Quadrant blocks (▖▗▘▝) | 4 (2×2) | 160×60 = 9,600 |
| Braille (⠁⠂⠃...⣿) | 8 (2×4) | 160×120 = 19,200 |
| Sixel protocol | Произвольное | До нативного разрешения терминала |

### B.3. 16 квадрантных паттернов — основа кодека .babe

Каждый квадрантный символ кодирует паттерн 2×2, где каждый субпиксель либо foreground, либо background:

```
Паттерн:  TL TR    Символ    Бинарное значение
          BL BR

0000 = ⬜⬜/⬜⬜ = (пробел)   = 0b0000
0001 = ⬜⬜/⬜⬛ = ▗          = 0b0001
0010 = ⬜⬜/⬛⬜ = ▖          = 0b0010
0011 = ⬜⬜/⬛⬛ = ▄          = 0b0011
0100 = ⬜⬛/⬜⬜ = ▝          = 0b0100
0101 = ⬜⬛/⬜⬛ = ▐          = 0b0101
0110 = ⬜⬛/⬛⬜ = ▞          = 0b0110
0111 = ⬜⬛/⬛⬛ = ▟          = 0b0111
1000 = ⬛⬜/⬜⬜ = ▘          = 0b1000
1001 = ⬛⬜/⬜⬛ = ▚          = 0b1001
1010 = ⬛⬜/⬛⬜ = ▌          = 0b1010
1011 = ⬛⬜/⬛⬛ = ▙          = 0b1011
1100 = ⬛⬛/⬜⬜ = ▀          = 0b1100
1101 = ⬛⬛/⬜⬛ = ▜          = 0b1101
1110 = ⬛⬛/⬛⬜ = ▛          = 0b1110
1111 = ⬛⬛/⬛⬛ = █          = 0b1111
```

Каждый символ: 4 бита (паттерн) + 16 бит (fg RGB565) + 16 бит (bg RGB565) = **36 бит**. Это и есть ядро кодека .babe.

## C. Формула Braille-рендеринга

Символы шрифта Брайля (⠁⠂⠃...⣿) — набор из **256 комбинаций** точек в сетке 2×4. Каждый символ кодирует **8 субпикселей**.

### C.1. Расположение точек

```
┌─────┐
│ 1  4│
│ 2  5│
│ 3  6│
│ 7  8│
└─────┘
```

### C.2. Формула Unicode codepoint

```
codepoint = 0x2800 + (dot1×1 + dot2×2 + dot3×4 + dot4×64 + dot5×8 + dot6×16 + dot7×32 + dot8×128)
```

### C.3. Пример

Если нужно закрасить точки 1, 3, 5, 7:
```
dot1=1, dot3=1, dot5=1, dot7=1
codepoint = 0x2800 + (1 + 4 + 8 + 32) = 0x2800 + 45 = 0x282D = ⠭
```

> **Релевантность для TVCP:** Braille-режим может быть добавлен как альтернатива квадрантным блокам для монохромных терминалов или для максимального разрешения (160×120 на 80×30 терминале). Однако Braille-символы не поддерживают раскраску верхней/нижней половины отдельно — только одноцветные.

## D. Проблема двойной конвертации

При тестировании конвертера мы обнаружили важный артефакт, который релевантен для дизайна кодека.

### D.1. Проблема

Если взять скриншот ASCII art и пропустить его через конвертер заново, качество **резко деградирует**.

### D.2. Причины

1. **Потеря исходной информации:** Скриншот ASCII art — это растровое изображение текстовых символов. Оригинальные пиксели уже утрачены.
2. **Несовпадение сеток:** Блоки конвертера не совпадают с границами символов в скриншоте → моирэ-эффект.
3. **Интерполяция при ресайзе:** Ресайз скриншота размывает чёткие границы символов.
4. **Артефакты шрифта:** Антиалиасинг шрифта на скриншоте создаёт серые пиксели, которые неверно интерпретируются.

### D.3. Вывод для дизайна кодека .babe

**Кодек должен передавать символьные данные, а не растровые.** Формат пакета для TVCP: `[glyph_index][fg_color][bg_color]`, а не JPEG/PNG-подобный поток. Это фундаментальное архитектурное решение — мы кодируем семантику (какой символ, какие цвета), а не пиксели.

## E. Zoom: технические детали из исследования

Данные собраны из веб-исследования для обоснования рыночной ниши TVCP.

### E.1. Архитектура Zoom

- Zoom использует **проприетарную реализацию H.264**, оптимизированную для десктопных приложений (macOS, Windows)
- Zoom Web SDK использует **WebAssembly-модули** вместо стандартных браузерных кодеков → больше CPU, чем нативный WebRTC
- Zoom Web SDK ограничен **720p** и часто показывает более низкое разрешение на реальных устройствах
- Аудиопоток Zoom **заблокирован на low-bandwidth speech** — нельзя настроить для музыки

### E.2. Точные данные по трафику Zoom

Источник: zoom.com, compareinternet.com (сентябрь 2025)

| Тип звонка | Потребление трафика |
|---|---|
| 1:1 SD (Standard Definition) | 540 MB/час |
| 1:1 720p HD | 1.08 GB/час |
| 1:1 1080p | 1.62 GB/час |
| Галерея 25 участников | 2.0 Mbps bandwidth |
| Галерея 49 участников | до 4.0 Mbps bandwidth |
| Screen share (только экран) | 50–75 kbps |
| Screen share + видео | 50–150 kbps |

### E.3. Минимальные требования Zoom

| Параметр | Значение |
|---|---|
| Минимальный канал (видео) | 3 Mbps download, 1.8 Mbps upload |
| HD видео | 3.8 Mbps download, 3.8 Mbps upload |
| Рекомендуемый CPU | Intel i5-10400 / AMD Ryzen 5 3600 |
| Рекомендуемая RAM | 16 GB DDR4 |
| Отличное соединение | Latency <50ms, Jitter <10ms, Packet loss <0.1% |

### E.4. Сравнение с TVCP

| Параметр | Zoom | TVCP |
|---|---|---|
| Минимальный канал | 1.8 Mbps | **200 kbps** (в 9× меньше) |
| Трафик/час | 540 MB – 1.62 GB | **~90 MB** (в 6–18× меньше) |
| RAM | 8–16 GB | **256 MB** (в 32–64× меньше) |
| CPU | i5+ | **Любой ARM** (Raspberry Pi Zero) |
| GUI | Обязательно | **Не нужен** |

## F. WebRTC vs Zoom: тест Jitsi на перегрузку сети

Источник: jitsi.org/blog (2021) — «A Simple Congestion Test for Zoom»

### F.1. Условия теста

- 1:1 видеозвонок через Zoom и WebRTC (Jitsi Meet) одновременно
- Первые 10 секунд — нормальное WiFi
- Секунда 10 — ограничение bandwidth до **500 kbps** (up + down)
- Замеры: время до частичной адаптации, полной адаптации, восстановления

### F.2. Результаты

| Метрика | WebRTC (Jitsi) | Zoom |
|---|---|---|
| Уведомление о проблемах | Быстро | Быстро |
| Частичная адаптация (видео на 500 kbps) | ~20 секунд | ~156 секунд |
| Полная адаптация | ~30 секунд | ~180+ секунд |
| Восстановление до SD (~1 Mbps) | Быстро | Быстро |
| Восстановление до HD (~2 Mbps) | ~10 секунд | ~30 секунд |

### F.3. Ключевой вывод для TVCP

**Zoom крайне плохо адаптируется к каналам ниже 500 kbps** — ему нужно почти 3 минуты. WebRTC адаптируется за 20 секунд. TVCP **нативно работает при 200 kbps** и не нуждается в адаптации — его формат изначально рассчитан на узкие каналы.

## G. Детальные сценарии использования с расчётами

### G.1. Сценарий: SSH-звонки при инцидентах

**Реальный кейс:** Ночной инцидент. SRE подключается к production-серверу через SSH из дома. Видит аномалию в логах.

**Без TVCP:** SSH → видит проблему → открывает Zoom → ждёт коллегу → коллега подключается → объяснение словами что видишь на экране → 5–10 минут потеряно.

**С TVCP:** SSH → видит проблему → `say "[yggdrasil-address]"` → 2 секунды → коллега видит то же самое + видеосвязь → инцидент решается в единой сессии.

**Экономический расчёт:** Средний инцидент = 47 минут (PagerDuty State of Digital Operations). 5 минут на setup Zoom = 10% от MTTR. При 50 инцидентах/месяц = 250 минут = 4.2 часа/месяц. При стоимости SRE $80/час = **$336/месяц экономии** только на setup time.

### G.2. Сценарий: IoT-видеомониторинг через сотовую сеть

**Расчёт трафика:**

| Решение | Трафик/час | Стоимость на сотовой (~$0.01/MB) | Стоимость/месяц (24/7) |
|---|---|---|---|
| TVCP (Say) | ~90 MB | $0.90 | **$648** |
| WebRTC 360p | ~450 MB | $4.50 | $3,240 |
| Zoom SD | ~540 MB | $5.40 | $3,888 |
| IP-камера (RTSP H.264) | ~350 MB | $3.50 | $2,520 |

**На спутниковом интернете (Iridium/Inmarsat, $1–5/MB):**

| Решение | Стоимость/час | Стоимость/месяц (8ч/день) |
|---|---|---|
| TVCP | $90–450 | $21,600–$108,000 |
| Zoom | $2,700–$8,100 | $648,000–$1,944,000 |

Разница: **6–18× экономии**. Для морских судов с ежемесячным бюджетом на связь $5,000–20,000 это определяющий фактор.

### G.3. Сценарий: Приватность и анонимность

**Модель угроз (расширенная):**

| Угроза | Zoom | Signal | TVCP + Yggdrasil | TVCP + Ygg + Tor |
|---|---|---|---|---|
| Сервер знает участников | ✓ | ✓ (телефоны) | ✗ (нет сервера) | ✗ |
| Логи у провайдера | Zoom CDN IPs | Signal IPs | Ygg overlay (видны только peering) | Только Tor guard |
| Аккаунт требуется | Email | Телефон | Крипто-ключ | Крипто-ключ |
| E2E шифрование | Опционально | Всегда | Всегда (Yggdrasil) | Двойное (Ygg + Tor) |
| Можно заблокировать | ✓ (домены) | ✓ (серверы) | Сложно (mesh) | Очень сложно |
| Работает через Tor | ✗ | ✗ | ✓ | ✓ (нативно) |
| Метаданные: кто с кем | Zoom знает | Signal знает | Никто не знает | Никто не знает |
| Метаданные: когда | Zoom знает | Signal знает | Никто не знает | Никто не знает |
| Метаданные: откуда | Zoom CDN видит IP | Signal видит IP | Ygg peer видит IP | Tor скрывает |

### G.4. Сценарий: Автономная mesh-сеть без интернета

**Конфигурация:**
- 3 ноутбука с WiFi
- 1 WiFi access point (роутер без интернета)
- Каждый ноутбук: Yggdrasil + TVCP

**Результат:** Полноценные видеозвонки между всеми участниками без единого байта интернет-трафика. Если один узел отключается, маршруты перестраиваются автоматически.

**Применения:**
- Военные учения — тактическая связь без зависимости от гражданской инфраструктуры
- Строительная площадка — WiFi mesh между вагончиками
- Фестивали/массовые мероприятия — сотовая сеть перегружена, но локальный mesh работает
- Подземные работы — шахты, тоннели, где нет покрытия
- Морские суда — внутрикорабельная связь

### G.5. Сценарий: DevOps-интеграция

**Примеры кода:**

```bash
# В .bashrc — алиасы для быстрого вызова

    say "[oncall-engineer-ygg-address]" &

# В Kubernetes CronJob — видеопроверка камеры

```

## H. Yggdrasil: расширенная техническая справка

### H.1. Основные характеристики

- **Тип:** Экспериментальная compact routing scheme, userspace software router
- **Сеть:** End-to-end зашифрованная IPv6 overlay-сеть
- **Адресация:** Префикс `200::/7` (mesh prefix)
- **Пиринг:** TCP/TLS через IPv4 или IPv6
- **Обнаружение:** Автоматический broadcast peer discovery на LAN-интерфейсах
- **Маршрутизация:** Self-arranging spanning tree + DHT
- **Авто-режим:** `yggdrasil --autoconf` — автоматический пиринг с узлами в той же подсети
- **Платформы:** Linux, macOS, Windows, FreeBSD, OpenBSD, OpenWrt

### H.2. Docker mesh fabric

Yggdrasil может автоматически формировать глобальную mesh-сеть между Docker-контейнерами, VM, серверами, объединяя всё в единый IPv6-сегмент:

```yaml
# docker-compose.yml — 3 узла Yggdrasil
version: '3'
services:
  node1:
    image: yggdrasil/yggdrasil
    command: yggdrasil --autoconf
    networks: [mesh]
  node2:
    image: yggdrasil/yggdrasil
    command: yggdrasil --autoconf
    networks: [mesh]
  node3:
    image: yggdrasil/yggdrasil
    command: yggdrasil --autoconf
    networks: [mesh]
networks:
  mesh:
    driver: bridge
```

### H.3. Контактная книга Say

Say хранит контакты как файловую систему:
```
contacts_dir/
├── Alice/
│   ├── laptop    (содержит IPv6 адрес)
│   └── phone     (содержит IPv6 адрес)
```

## I. Исходный анализ статьи Ваничкина на Habr

Источник: habr.com/ru/articles/973868/ — Сергей Ваничкин

### I.1. Ключевые технические решения автора

1. **Кодек .babe** (Bi-Level Adaptive Block Encoding): собственный формат сжатия, специально заточенный под блочную графику
2. **Квадрантная кластеризация:** Для каждого блока 2×2 пикселя — k-means с k=2, подбор оптимального глифа из 16 вариантов
3. **Go implementation:** Единый бинарник без зависимостей, кросс-компиляция
4. **Yggdrasil P2P:** Embedded Yggdrasil node в Go-бинарнике
5. **G.722 аудио:** Чистая Go-реализация, без cgo

### I.2. Производительность .babe vs существующие форматы

По данным автора:
- **Скорость кодирования:** Быстрее QOI и JPEG
- **Размер файла:** Между QOI (больше) и JPEG (меньше)
- **Блочная структура:** Идеально ложится на потоковое видео — можно кодировать кадр по частям

### I.3. Что автор не реализовал (возможности для расширения)

- Межкадровое сжатие (P-frames) — только I-frames
- Групповые звонки
- Мобильные клиенты
- Screen sharing
- Sixel/Kitty высококачественный режим
- Adaptive bitrate при изменении качества канала
- Шумоподавление аудио

### J.2. Стратегия fallback

```
1. Проверить TERM, COLORTERM, TERM_PROGRAM
2. Если COLORTERM=truecolor → 24-bit + квадрантные блоки
3. Если TERM=xterm-256color → 256 colors + half-blocks
4. Если TERM=xterm → 16 colors + стандартные символы ramp
5. Если ничего → чистый ASCII (@#*+-:. )
6. Если Sixel поддерживается → предложить --sixel режим
7. Если Kitty Graphics → предложить --kitty режим
```

### J.3. Определение размера изображения

```go
// Размер терминала в символах
cols, rows := getTerminalSize() // ioctl TIOCGWINSZ

// Эффективное разрешение видео
videoWidth := cols              // 1 символ = 1 «пиксель» по ширине
videoHeight := rows * 2         // 1 символ = 2 «пикселя» по высоте (▀▄)

// С квадрантами
videoWidth := cols * 2          // 1 символ = 2 субпикселя по ширине
videoHeight := rows * 2         // 1 символ = 2 субпикселя по высоте

// Стандартный терминал 80×24:
// Half-blocks: 80×48 = 3,840 эффективных пикселей
// Quadrants:   160×48 = 7,680 эффективных пикселей
```
