# Перекрёстное обогащение: TVCP × Doom Over DNS

## Введение

Два проекта используют разные подходы к передаче данных в терминальной среде:

- **TVCP** (`/home/user/infon`): real-time UDP видеозвонки с кастомным BABE-кодеком
- **Doom Over DNS** ([статья](https://habr.com/ru/news/1015672/)): доставка игры (~4 MB) через DNS TXT-записи

Данный документ — техническая матрица взаимного обогащения: что каждый проект может
взять у другого, с расчётами и конкретными точками интеграции.

---

## Часть 1: Что TVCP берёт у Doom Over DNS

### 1.1 DNS TXT как глобальная книга контактов (Peer Discovery)

**Проблема:** для звонка нужно знать Yggdrasil IPv6-адрес (`[200:abc::1]:5000`).

**Решение:** DNS TXT-запись хранит адрес пользователя:

```
_tvcp.alice.example.com.  TXT  "ygg=200:abc:def::1:5000 v=1 pubkey=a1b2c3..."
```

```bash
# Было:
tvcp call [200:abc:def::1]:5000

# Станет:
tvcp call alice@example.com   # клиент делает DNS lookup автоматически
```

Подробное описание: [`docs/dns-peer-discovery.md`](dns-peer-discovery.md)

---

### 1.2 DNS SRV для bootstrap STUN/TURN

**Проблема:** серверы NAT-traversal жёстко зашиты в конфигурации.

**Решение:** стандартные DNS SRV-записи:

```
_stun._udp.tvcp.example.com.  SRV  0 0 3478 stun.example.com.
_turn._udp.tvcp.example.com.  SRV  0 0 3478 turn.example.com.
```

При запуске TVCP выполняет DNS SRV lookup и автоматически находит серверы.
Аналог: WebRTC использует именно этот механизм.

**Точка интеграции:** `internal/stun/stun_client.go`

---

### 1.3 DNS как fallback-канал сигнализации

**Сценарий:** корпоративный файрвол блокирует UDP, но DNS (порт 53) открыт.

**Механизм:**
```
Сторона А                    DNS-сервер               Сторона Б
    │                             │                         │
    ├──CALL_REQUEST (TXT)────────►│                         │
    │                             │◄────────POLL (TXT)──────┤
    │                             ├──CALL_ACCEPT (TXT)─────►│
    │◄══════ Media: UDP/Yggdrasil ════════════════════════► │
```

DNS tunneling (~40 Kbps) — только для сигнализации (несколько сотен байт),
не для медиапотока.

**Ограничение:** DNS-запросы логируются провайдером → сниженная конфиденциальность.
Рекомендуется только как последний резерв при заблокированном UDP.

---

### 1.4 Паттерн checksum-фрагментации для file sharing

Doom Over DNS разбивает ~4 MB на ~1964 TXT-фрагмента, каждый с checksum,
с поддержкой докачки. В TVCP `experimental/file_sharing` уже использует 64 KB чанки
с SHA-256 верификацией — тот же паттерн.

**Применение:** при прерванной передаче файла — возобновление с последнего
верифицированного чанка. Уже частично реализовано; доработка потребует
сохранения состояния передачи на диск.

**Точка интеграции:** `experimental/file_sharing/`

## Часть 2: Что Doom Over DNS берёт у TVCP

### 2.1 BABE-кодек: 20× сжатие для DNS-хранилища

**Проблема Doom Over DNS:** 4 MB WAD-файла требует ~1964 TXT-записей.
Бесплатный тир Cloudflare — только 185 записей.

**Идея:** перед упаковкой в DNS TXT-записи прогнать игровые кадры через
BABE-кодек TVCP:

```
Doom raw frame (320×200 RGB):       192,000 байт
BABE frame (80×40 блоков):           32,000 байт   ÷6
BABE frame + zlib:                    ~4,000 байт   ÷48

Целый WAD (4 MB raw):
  Без BABE: ~1,964 TXT-записей
  С BABE:   ~40 TXT-записей (в рамках бесплатного тира!)
```

Конечно, WAD — не только кадры, но и логика. BABE применима к **видео-пакету**
в составе WAD (демо-файлы, заставки). Для самой игровой логики сжатие иное.

---

### 2.2 P-frame дельты для анимаций

TVCP хранит только **изменившиеся блоки** между кадрами (P-фреймы):

```
Типичный Doom: 50–70% экрана меняется при движении
P-frame (50% изменений): ~9,600 байт вместо 32,000 байт I-фрейма

Для DNS: 38 TXT-записей вместо 125 на анимированную заставку
```

Это применимо к **демо-записям Doom** (`.lmp` файлы) — их можно хранить
как последовательность дельта-кадров в DNS TXT вместо сырого формата.

---

### 2.3 UDP P2P вместо Cloudflare DNS CDN

**Проблема Doom Over DNS:** зависимость от Cloudflare для глобального кэша.

**Альтернатива:** использовать Yggdrasil P2P для раздачи игровых данных — аналог
BitTorrent, но поверх Yggdrasil mesh сети:

```
Узел А (имеет Doom) ←──Yggdrasil──→ Узел Б (запрашивает)
                    ←──Yggdrasil──→ Узел В (тоже раздаёт)
```

Преимущества: нет зависимости от Cloudflare, нет ограничений на количество записей,
скорость — мегабайты/сек вместо килобайт/сек DNS.

**Точка интеграции:** `internal/yggdrasil/` + `experimental/file_sharing/`

## Часть 3: Расчёт пропускной способности

### DNS vs UDP: сравнение для разных задач

| Задача | DNS (Doom Over DNS) | UDP (TVCP) | Победитель |
|--------|-------------------|-----------|-----------|
| Загрузка игры (4 MB) | ~800 сек | ~11 сек | UDP ×72 |
| Peer discovery (100 байт) | ~0.05 сек | ~0.01 сек | DNS (firewall bypass) |
| Видео 15 FPS (350 KB/s) | Невозможно | ✅ Норма | UDP |
| Сигнализация (500 байт) | ~0.1 сек | ~0.01 сек | Оба работают |
| Хранение данных (постоянно) | ✅ DNS TTL-кэш | ❌ Не предназначен | DNS |

### Расчёт: DNS-видеозвонок (теоретически)

```
Один BABE I-фрейм (80×30):         24,000 байт
После zlib:                         ~3,000 байт
DNS TXT-записей нужно:              ~12 записей (по 255 байт)
Время загрузки при 20 req/сек:      ~0.6 сек

→ Теоретический FPS через DNS: ~1.7 FPS
→ Практический (с задержками): ~0.2–0.5 FPS

Вывод: «слайд-шоу» раз в 2–5 секунд — не видеозвонок, но технически работает
```

## Итоговые выводы

### Три реалистичных гибридных сценария

**Сценарий 1: «Смотрим игру вместе»** (уже возможно)
```bash
# Игрок А запускает игру + расшаривает экран
tvcp share [200:alice::1]:5000  # или: tvcp share alice@example.com

# Игрок Б смотрит в своём терминале в 15 FPS
# Оба общаются голосом через тот же звонок
```

**Сценарий 2: «Doom в DNS → TVCP трансляция»** (нужна интеграция)
```
DNS TXT (Doom загрузка) → BABE codec → TVCP P2P трансляция
Загрузка: <40 TXT-записей (бесплатный тир Cloudflare!)
Трансляция: 15 FPS, ~50 KB/s
```

**Сценарий 3: «Звони по имени, играй вместе»** (нужна разработка)
```bash
tvcp call alice@example.com  # DNS peer discovery
# → автоматически находит Yggdrasil-адрес Алисы
# → устанавливает P2P соединение
# → запускает совместную пошаговую игру (UNO, шахматы)
```

### Ключевой инсайт

DNS и UDP решают **разные временны́е задачи**:
- **DNS**: хранение и обнаружение (минуты, часы, постоянно)
- **UDP**: передача в реальном времени (миллисекунды)

Оптимальная архитектура **использует оба**: DNS для bootstrapping и peer discovery,
UDP для фактической передачи медиа и игрового состояния.
