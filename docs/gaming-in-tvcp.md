# Игры в TVCP: архитектура и возможности

## Обзор

TVCP изначально создавался как платформа для видеозвонков, однако его архитектура
допускает запуск и демонстрацию игр. В директории `experimental/` уже реализованы
5 игр. Данный документ описывает техническую базу, ограничения и пути развития.

---

## 1. Уже реализованные игры (`experimental/`)

| Игра | Тип | Механизм синхронизации |
|------|-----|----------------------|
| Tic-Tac-Toe | Пошаговая, 2 игрока | JSON через `PacketTypeTextChat` |
| UNO | Пошаговая, 2–4 игрока | JSON через `PacketTypeTextChat` |
| Trivia | Викторина, 1–N игроков | JSON через `PacketTypeTextChat` |
| Wordle | Одиночная / совместная | JSON через `PacketTypeTextChat` |
| Hangman | Одиночная / совместная | JSON через `PacketTypeTextChat` |

Все существующие игры — **пошаговые**. Их состояние передаётся в виде JSON-сообщений
через стандартный текстовый канал. Это работает при любой задержке сети.

## 2. BABE-кодек как движок рендеринга

### Принцип работы

TVCP конвертирует произвольное изображение (`image.Image`) в сетку Unicode-блоков:

```
Входное изображение (любое разрешение)
          ↓
  Масштабирование под размер терминала
          ↓
  Разбивка на ячейки 2×2 пикселя
          ↓
  Для каждой ячейки: k-means кластеризация (2 цвета)
          ↓
  Выбор символа из 16 квадрантных блоков (▀▄▌▐▖▗▘▝…)
          ↓
  Блок: символ + цвет переднего плана + цвет фона
          ↓
  ANSI-вывод в терминал
```

Ключевой метод: `internal/codec/babe/converter.go` → `ImageToFrame(img, width, height)`

### Характеристики рендеринга

| Параметр | Значение |
|----------|---------|
| Пиксельное разрешение | 2 пикселя на блок → 80×30 терминал = **160×60 px** |
| Цветовая глубина | 24-bit RGB (fg + bg независимо) |
| Глиф-сет | 16 квадрантных блоков + полный ASCII |
| FPS (шаринг экрана) | 15 кадров/сек |
| Сжатие P-фреймами | только изменившиеся блоки (~14% от I-фрейма) |

### Сравнение с разрешениями игровых платформ

```
TVCP (80×30 терминал):    160×60 px    ← текущий стандарт
TVCP (80×50 терминал):    160×100 px   ← широкие терминалы
Game Boy (1989):           160×144 px
Game Boy Color (1998):     160×144 px
NES (1983):                256×240 px
Doom shareware (1993):     320×200 px
```

Вывод: разрешение TVCP сопоставимо с **Game Boy Color**. Для пошаговых игр и
медленных платформеров достаточно, для динамичных 3D-игр (Doom) — низковато.

## 3. Шаринг игры через `tvcp share`

### Как это работает сейчас

Самый простой способ «играть через TVCP» уже доступен:

```bash
# Терминал 1: запустить игру (например, nethack, или любую TUI-игру)
nethack

# Терминал 2: расшарить терминал собеседнику
tvcp share [200:abc::1]:5000

# Собеседник получает живой поток терминала в 15 FPS
```

Технически: `internal/screen/screen_share.go` захватывает вывод процесса через
псевдо-терминал (PTY), конвертирует в `terminal.Frame`, передаёт через
`PacketTypeScreen` поверх UDP.

### Ограничение: только просмотр

Текущая реализация — **однонаправленная**. Собеседник видит игру, но не может
управлять ею. Это аналог стриминга (Twitch), но в терминале.

```
Игрок А                         Игрок Б
  Doom/Nethack запущен  ──────►  Видит экран в 15 FPS
  Управляет локально             Не может управлять
  Голосовой звонок ◄────────►   Голосовой звонок
```

## 4. Отображение произвольной графики: Doom через BABE

### Техническая осуществимость

Doom выводит кадры в `image.RGBA` (через SDL или framebuffer). BABE-кодек принимает
любой `image.Image`. Следовательно, цепочка технически возможна:

```go
// Концептуальный код (не реализован)
doomFrame := doom.GetCurrentFrame()        // image.RGBA, 320×200
termFrame := babe.ImageToFrame(doomFrame, 80, 40) // terminal.Frame, 80×40 блоков
tvcp.SendFrame(termFrame)                  // UDP → собеседники
```

### Что получится визуально

```
Оригинал Doom (320×200):         Через TVCP BABE (160×80 px):
┌────────────────────┐           ┌──────────────────┐
│ Детализированный   │           │ Пиксельный,      │
│ спрайт врага       │    →      │ угловатый враг   │
│ 256 цветов         │           │ ~16 узнаваемых   │
│                    │           │ цветовых зон     │
└────────────────────┘           └──────────────────┘
```

Узнаваемость сохраняется (структура кадра различима), детали теряются.
Эффект «демейка» — ретро-стиль напоминает порты для Game Boy.

### Расчёт пропускной способности

```
Doom raw I-фрейм (320×200 RGB):     192,000 байт
BABE I-фрейм (80×40 блоков):          32,000 байт  (в 6× меньше)
BABE I-фрейм + zlib:                  ~5,000 байт  (в 38× меньше)
BABE P-фрейм (типичный бой, ~30%):    ~3,360 байт

При 15 FPS: ~50 KB/s видео (vs 350 KB/s TVCP-видео с камеры)
```

BABE-рендеринг Doom по сети дешевле, чем стриминг реальной камеры.

## 5. Архитектура co-op игр в реальном времени

### Проблемы текущей архитектуры

Для **real-time игр** (даже простых, с вводом) текущий протокол TVCP не подходит:

| Проблема | Значение | Порог для игр |
|---------|---------|--------------|
| Джиттер-буфер | 50–500 мс | Платформер: <100 мс |
| Типичный RTT (Yggdrasil) | 100–300 мс | Шутер: <30 мс |
| Нет канала ввода | — | Нужен `PacketTypeGameInput` |

### Что реалистично

| Тип игры | Требование к latency | Реализуемость в TVCP |
|----------|---------------------|---------------------|
| Пошаговые (шахматы, UNO) | Любая | ✅ Уже реализовано |
| Медленные стратегии (Nethack) | <500 мс | ✅ Через screen sharing |
| Платформеры (Mario-стиль) | <100 мс | ⚠️ На грани (Yggdrasil LAN) |
| Шутеры (Doom co-op) | <30 мс | ❌ Невозможно через Yggdrasil WAN |

### Схема добавления `PacketTypeGameInput`

Для пошаговых и медленных игр можно добавить новый тип пакета:

```go
// В internal/network/packet.go добавить:
PacketTypeGameInput uint8 = 0x08  // Игровой ввод

// Структура:
type GameInputPacket struct {
    GameID    uint32   // Идентификатор сессии игры
    Timestamp uint64   // Время нажатия (мс)
    InputType uint8    // 0=keydown, 1=keyup, 2=gamestate
    KeyCode   uint32   // Код клавиши
    Payload   []byte   // Доп. данные (JSON состояние для пошаговых)
}
```

Для пошаговых игр это достаточно: состояние передаётся целиком при каждом ходу,
латентность не критична.

## 6. Сравнение: Doom Over DNS vs TVCP для игр

| Критерий | Doom Over DNS | TVCP + BABE |
|----------|--------------|-------------|
| **Загрузка игры** | Из DNS TXT-записей | Не применимо |
| **Трансляция кадров** | Невозможно (DNS слишком медленный) | 15 FPS @ ~50 KB/s |
| **Ввод игрока** | Локальный | Через `PacketTypeGameInput` (в разработке) |
| **Разрешение** | Нативное (320×200) | 160×80 px (terminal) |
| **Совместная игра** | ❌ | ✅ Просмотр (шаринг), ✅ Пошаговые |
| **Firewall** | DNS проходит везде | UDP нужен |
| **Сжатие** | zlib по фрагментам | BABE + P-фреймы |

### Гибридный сценарий: лучшее из двух миров

```
1. Загрузка игры: через DNS TXT-записи (как в Doom Over DNS)
                          ↓
2. Запуск локально: Doom / любая игра
                          ↓
3. Трансляция: TVCP BABE кодек → UDP → собеседники
                          ↓
4. Общение: голосовой/видеозвонок поверх того же соединения
```

Это объединяет:
- **доставку данных без сервера** (DNS CDN)
- **реальное время** (TVCP UDP)
- **terminal-native рендеринг** (BABE)
